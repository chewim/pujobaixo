PUJOBAIXO — Cómo funciona el despliegue (GitHub Pages)
========================================================

Desde julio de 2026 este sitio se sirve desde GitHub Pages, no desde
Netlify (Netlify agotó los créditos gratuitos del plan y pausó los
despliegues).

Qué contiene el repositorio:
- index.html         -> app
- 404.html            -> copia idéntica de index.html. GitHub Pages no
                         soporta el archivo _redirects de Netlify para
                         rutas tipo SPA (ej. /auth/callback). El truco:
                         cuando alguien visita una ruta que no es un
                         archivo real, GitHub Pages sirve 404.html en
                         su lugar SIN cambiar la URL del navegador, así
                         que la app arranca igual y detecta la ruta
                         correctamente.
- CNAME               -> contiene "pujobaixo.cat", le dice a GitHub
                         Pages qué dominio propio usar.
- support.js          -> runtime de la app
- supabase.umd.js     -> librería de Supabase alojada localmente
- assets/             -> imágenes

CÓMO SE DESPLIEGA A PARTIR DE AHORA
Cada vez que se hace `git push` a la rama `main` de
github.com/chewim/pujobaixo, GitHub Pages publica los cambios solo,
normalmente en menos de un minuto. No hace falta arrastrar zips ni
tocar Netlify.

CONFIGURACIÓN QUE YA DEBERÍA ESTAR HECHA (una sola vez)
1. En github.com/chewim/pujobaixo → Settings → Pages:
   - Source: "Deploy from a branch"
   - Branch: main / (root)
   - Custom domain: pujobaixo.cat
   - "Enforce HTTPS" activado (tarda unos minutos en aparecer la casilla
     tras configurar el DNS)

2. En IONOS (DNS del dominio pujobaixo.cat), sustituir los registros
   que apuntaban a Netlify por estos (los que pide GitHub Pages):
   - 4 registros A en la raíz (@) apuntando a:
       185.199.108.153
       185.199.109.153
       185.199.110.153
       185.199.111.153
   - 1 registro CNAME para "www" apuntando a:
       chewim.github.io

No hace falta tocar nada en Supabase ni en el DNS de Resend (DKIM/SPF/
DMARC): son registros aparte y no dependen de dónde se aloje la web.
